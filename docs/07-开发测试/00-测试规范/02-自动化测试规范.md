# 自动化测试规范

## 1. 测试金字塔

```
        ┌─────────┐
        │   E2E   │  少量关键用户旅程
       ┌┴─────────┴┐
       │  集成测试  │  模块间交互
      ┌┴───────────┴┐
      │   单元测试   │  大量快速反馈
     ┌┴─────────────┴┐
     │     CI/CD      │
```

## 2. CI/CD集成要求

### 2.1 必须自动化的测试

| 测试类型 | 触发时机 | 失败策略 |
|----------|----------|----------|
| 单元测试 | 每次PR | 必须通过 |
| 集成测试 | 每次PR | 必须通过 |
| Lint/Type Check | 每次PR | 必须通过 |
| 安全扫描 | 每次PR | 警告+必须修复 |

### 2.2 自动化通过标准

- 单元测试覆盖率 ≥ 80%
- 全部P0用例通过
- 无S0/S1级别新缺陷

## 3. 单元测试规范

### 3.1 测试范围

**必须覆盖**:
- 所有业务逻辑函数
- 数据转换/计算逻辑
- 权限校验逻辑
- 异常处理分支

**建议覆盖**:
- 工具函数
- 组件渲染逻辑

### 3.2 命名规范

```javascript
// 被测试文件: src/utils/formatCurrency.js
// 测试文件:   src/utils/__tests__/formatCurrency.test.js

describe('formatCurrency', () => {
  it('should format CNY correctly', () => {
    expect(formatCurrency(170000000000)).toBe('¥1,700亿');
  });

  it('should handle decimal values', () => {
    expect(formatCurrency(284000000.56)).toBe('¥2.84亿');
  });

  it('should throw on invalid input', () => {
    expect(() => formatCurrency('abc')).toThrow();
  });
});
```

## 4. 集成测试规范

### 4.1 API测试

```javascript
// 使用Supertest
const request = require('supertest');

describe('Workbench API', () => {
  it('GET /api/v1/workbench/kpi - should return kpi data', async () => {
    const res = await request(app)
      .get('/api/v1/workbench/kpi')
      .set('Authorization', `Bearer ${token}`)
      .expect(200);

    expect(res.body.data).toHaveProperty('managedScale');
    expect(res.body.data).toHaveProperty('occupancyRate');
  });
});
```

### 4.2 数据库测试

- 每个测试用例前清空相关表
- 使用事务回滚确保隔离
- 禁止在测试中操作生产数据

## 5. E2E测试规范

### 5.1 核心用户旅程

| 旅程 | 覆盖功能 |
|------|----------|
| D1: 看总览 | 工作台KPI → 趋势图 → 异常摘要 |
| D2: 看异常 | 预警列表 → 详情 → 处置 |
| D3: 看穿透 | 项目组合 → 项目详情 → 指标钻取 |
| D4: 导出报告 | 筛选 → 生成 → 审批 → 导出 |

### 5.2 工具选择

- **Web**: Playwright / Cypress
- **API**: Postman / Newman
- **移动端**: Appium
