# UAT测试用例编写规范

## 1. UAT用例与开发用例的区别

| 维度 | 开发测试用例 | UAT测试用例 |
|------|--------------|-------------|
| 编写者 | 开发工程师 | 业务人员/实施顾问 |
| 关注点 | 技术正确性 | 业务价值实现 |
| 表述方式 | 技术术语 | 业务语言 |
| 验证内容 | 功能/接口/性能 | 场景/流程/体验 |

## 2. UAT用例结构

```
模块: [业务模块名称]
场景: [业务场景描述]
前置条件: [执行前需满足的业务状态]
测试步骤: [业务操作步骤（带截图指引）]
预期结果: [业务人员期望的结果]
验证方式: [如何验证（人工/对比数据）]
优先级: [P0/P1/P2]
```

## 3. 用例ID编码规则

```
UAT-{模块}-{序号}

模块编码:
- WB  : 工作台
- BI  : 经营分析
- PF  : 项目组合
- INT : 情报中心
- KB  : 知识库
- RG  : 风险预警
- CP  : Copilot
- ADM : 系统管理

示例:
- UAT-WB-001: 工作台-001号用例
- UAT-BI-001: 经营分析-001号用例
```

```markdown

**场景描述**:
[用业务语言描述这个场景要验证什么]

**前置条件**:
- [条件1]
- [条件2]

**测试步骤**:
1. [步骤1，包含操作位置指引]
2. [步骤2]
3. [步骤3]

**预期结果**:
- [结果1]
- [结果2]

**验证方式**:
[如何验证这个结果是正确的]

**优先级**: P0/P1/P2

**注意事项**:
[如有特殊要求或注意点]

**关联需求**: [对应的需求编号]
```

## 5. 用例编写示例

```markdown
### UAT-WB-001: 查看工作台核心指标

**场景描述**:
作为高管，我希望在工作台一眼看到核心经营指标，以便快速了解公司运营状况。

**前置条件**:
- 用户已登录系统
- 系统已完成数据同步
- 当日有最新数据

**测试步骤**:
1. 打开系统首页（工作台）
2. 找到页面顶部的KPI卡片区域
3. 查看各卡片显示的数值

**预期结果**:
- 在管规模显示约 ¥1,700亿（签约规模）
- 组合出租率显示约 92.4%
- NOI显示约 ¥2.84亿
- 风险预警显示红色3条、黄色7条
- 各指标有同比/环比变化标签

**验证方式**:
对比财务系统/运营系统中的同期数据，误差应<1%

**优先级**: P0

**注意事项**:
- 数据为脱敏测试数据，与实际数据可能不同

**关联需求**: M0-工作台-001
```

## 6. P0/P1/P2优先级定义

| 优先级 | 含义 | 示例 |
|--------|------|------|
| P0 | 核心业务功能，必须通过 | 登录、数据展示、预警通知 |
| P1 | 重要业务功能，影响效率 | 筛选搜索、导出功能 |
| P2 | 辅助功能，优化体验 | 快捷按钮、界面样式 |

## 7. 测试结果记录

执行测试时，记录以下信息：

| 字段 | 说明 |
|------|------|
| 执行人 | 测试执行人员姓名 |
| 执行时间 | 执行的日期时间 |
| 执行结果 | ✅ 通过 / ❌ 失败 / ⏸️ 阻塞 / ⏭️ 跳过 |
| 失败说明 | 如失败，描述具体问题 |
| 截图证据 | 失败时的截图 |
