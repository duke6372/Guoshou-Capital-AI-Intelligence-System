# UAT测试管理办法

## 1. UAT测试目的

UAT（User Acceptance Testing）用户验收测试旨在验证系统功能满足业务需求，确保系统在正式上线前达到预期使用标准。

## 2. UAT测试范围

### 2.1 纳入UAT的功能

| 模块 | 测试范围 |
|------|----------|
| 工作台 | KPI展示、异常摘要、待办事项 |
| 经营分析 | 指标查看、趋势分析、导出 |
| 项目组合 | 项目查看、对标、详情穿透 |
| 情报中心 | 情报查看、订阅、复核 |
| 知识库 | 知识查询、合同解析 |
| 风险预警 | 预警查看、处置、规则配置 |
| Copilot | 智能问答、报告生成 |
| 系统管理 | 权限配置、用户管理 |

### 2.2 UAT与开发测试的区分

| 测试内容 | 开发测试 | UAT测试 |
|----------|----------|----------|
| 单元测试 | ✅ 开发负责 | ❌ |
| API接口测试 | ✅ 开发负责 | ❌ |
| 组件功能测试 | ✅ 开发负责 | ❌ |
| 数据准确性验证 | ✅ 开发负责 | ✅ 客户确认 |
| 业务场景验证 | ❌ | ✅ 客户负责 |
| 指标口径确认 | ❌ | ✅ 数据Owner |
| 预警规则验证 | ❌ | ✅ 风控人员 |
| 权限配置验证 | ✅ | ✅ 双方确认 |

## 3. UAT测试准备

### 3.1 环境准备

| 环境 | 用途 | 数据要求 |
|------|------|----------|
| UAT环境 | 客户验收测试 | 脱敏后的生产级数据 |
| 测试账号 | 按角色分配 | 已配置对应权限 |

### 3.2 测试数据准备

**数据要求**:
- 测试数据必须脱敏，不含真实客户信息
- 数据量需覆盖主要业务场景
- 数据状态需包含正常/异常/边界情况

**数据准备清单**:
- [ ] 项目数据（至少5个典型项目）
- [ ] 指标数据（12个月历史数据）
- [ ] 预警数据（红/黄/提示级别）
- [ ] 情报数据（各类别至少3条）
- [ ] 知识数据（各分类至少5条）
- [ ] 合同数据（租赁/融资各1份）

### 3.3 人员准备

| 角色 | 人数 | 职责 |
|------|------|------|
| 业务人员 | 3-5人 | 执行业务场景测试 |
| 数据Owner | 1-2人 | 验证数据口径 |
| 风控人员 | 1-2人 | 验证预警规则 |
| 实施工程师 | 1-2人 | 技术支持 |

## 4. UAT测试执行

### 4.1 测试执行流程

```
Day 1: 启动会 + 培训
├── 介绍测试范围和流程
├── 演示系统基本操作
└── 分发测试账号和手册

Day 2-3: 业务场景测试
├── 按模块分组执行
├── 每日站立会议同步进度
└── 即时沟通问题

Day 4: 问题回归 + 补充测试
├── 验证问题修复
├── 补充遗漏测试点
└── 收集改进建议

Day 5: 验收评审
├── 测试结果汇报
├── 问题复盘
└── 签署验收意见
```

### 4.2 测试执行要求

1. **测试记录**: 每条用例需记录执行结果（通过/失败/阻塞）
2. **问题提交**: 发现问题立即在Jira提交，标注详细复现步骤
3. **截图留证**: 关键步骤需截图，作为问题佐证
4. **进度同步**: 每日同步测试进度和 blocker

### 4.3 测试分工建议

| 角色 | 负责模块 | 测试重点 |
|------|----------|----------|
| 高管用户 | 工作台、BI | KPI准确性、报告生成 |
| 运营人员 | 项目组合、情报 | 日常操作便捷性 |
| 风控人员 | 风险预警 | 预警准确性、处置流程 |
| 知识管理员 | 知识库 | 知识检索准确性 |
| 系统管理员 | 管理后台 | 权限配置正确性 |

## 5. UAT问题管理

### 5.1 问题提交规范

提交问题时需包含：
- **问题标题**: 简洁描述问题
- **问题描述**: 详细描述操作步骤
- **预期结果**: 期望的正确行为
- **实际结果**: 观察到的错误行为
- **截图/录屏**: 复现证据
- **问题等级**: 严重/一般/轻微
- **影响范围**: 受影响的用户范围

### 5.2 问题处理流程

```
客户提交 → 实施团队确认 → 开发团队修复
    ↓                            ↓
回归测试 ← 问题修复 ← 开发团队修复
    ↓
验证通过 → 问题关闭
    ↓
验证失败 → 重新打开
```

### 5.3 修复时效要求

| 问题等级 | 响应时间 | 修复时限 |
|----------|----------|----------|
| 严重 | 2小时内 | 24小时内 |
| 一般 | 4小时内 | 3天内 |
| 轻微 | 1天内 | 下版本 |

## 6. UAT验收标准

### 6.1 测试通过条件

| 优先级 | 要求 |
|--------|------|
| P0用例 | 100%通过 |
| P1用例 | ≥95%通过 |
| P2用例 | ≥80%通过 |

### 6.2 验收签字

测试完成后，需由以下人员签署验收意见：

| 角色 | 签字内容 |
|------|----------|
| 业务负责人 | 功能验收通过 |
| 数据Owner | 数据口径认可 |
| 风控负责人 | 风险规则认可 |
| 项目经理 | 整体验收确认 |

## 7. 注意事项

1. **测试环境**: UAT环境仅供测试，禁止直接操作生产数据
2. **账号安全**: 保管好测试账号，不要泄露给他人
3. **数据保护**: 测试中发现的数据问题不要外传
4. **沟通渠道**: 测试期间保持沟通畅通，即时反馈问题
