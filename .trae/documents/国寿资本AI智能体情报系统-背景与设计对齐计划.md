## Proposed Changes
### 1) 新增“材料抽取与分析”产物目录
新增目录与文件（用于沉淀可复用的分析资产）：
- `docs/source/`：存放从 PDF/PPTX 抽取出的原始文本（按文件名分文件）
- `docs/analysis/`：存放每份材料的结构化卡片与跨材料综合结论
- `docs/design/设计对齐稿.md`：最终交付的设计对齐稿（文字版）
- `scripts/extract_docs.py`：离线抽取脚本（PDF、PPTX → txt/markdown）

### 2) 自动抽取方案（PDF/PPTX → 文本）
实现策略（按优先级降级，确保可执行）：
1. **优先走纯 Python**（更可控、可跨环境）  
   - PDF：使用 `pypdf` 或 `pdfplumber` 抽取文本（先尝试直接抽取；如遇到扫描件则记录“需OCR”）
   - PPTX：使用 `python-pptx` 抽取每页标题/正文/备注
2. **如纯 Python 抽取效果差**，再尝试系统可用的命令行工具（若存在）  
   - PDF：`pdftotext`（若本机已安装）
3. **若识别为扫描件**（文本极少/乱码），再引入 OCR（作为可选增强，不阻塞第一版对齐稿）  
   - OCR 候选：Tesseract / macOS Vision（以“最轻量可用”为原则）

抽取标准化输出字段：
- 文本内容 + 基本元信息（源文件名、页码/页序、抽取方式、抽取质量评分：高/中/低）

### 3) 材料结构化分析（每份材料一张“要点卡”）
对每份材料输出 `docs/analysis/<材料名>.md`，统一模板：
- 这份材料在解决什么问题（目的）
- 覆盖的业务域/对象（范围）
- 关键结论/事实（可引用原文段落）
- 明确的需求/能力点（用户故事/功能点）
- 数据与系统边界（数据源、上下游、依赖）
- 约束与风险（合规、权限、口径、时效）
- 未决问题（需要向业务/技术确认）

### 4) 跨材料综合：形成“项目背景与问题定义”
在 `docs/analysis/00-背景综合.md` 里输出：
- 业务目标（为什么做）
- 目标用户与组织角色（谁来用）
- 关键场景与价值链（用来做什么决策/动作）
- 当前痛点与机会点（现状→目标态差距）
- 关键名词与口径（情报、知识、合同、项目、维度等）
- 成功指标（可度量：效率/覆盖/质量/合规）
- 约束与假设（必须成立的前提）

### 5) 设计对齐稿（文字版）
输出 `docs/design/设计对齐稿.md`，用于与设计/研发对齐，包含：
- 产品定位与北极星目标
- 角色与权限模型（最小可行：管理员/分析员/业务人员/审计等，最终以材料为准）
- 信息架构（一级/二级导航、全局搜索、工作台）
- 核心对象模型（情报条目、实体/关系、项目、合同、知识条目、数据指标等——以材料抽取为准）
- 关键用户旅程（从“提出问题→检索→聚合→研判→输出→留痕/审计”）
- 关键页面清单 + 交互说明（文字版）  
  - 工作台（待办/关注/订阅/告警）
  - 情报检索与过滤（维度、时间、来源、可信度）
  - 情报详情（证据链、关联实体、时间线、引用与导出）
  - 关系图谱/地图/时间线（如材料提及）
  - 知识库/合同库/项目库（结构化字段、版本、权限）
  - 研判报告生成与协作（模板、引用、审批、留痕）
  - 数据看板（口径、刷新频率、指标释义）
- 非功能需求（性能、可用性、可追溯、权限隔离、数据脱敏）
- 运营与增长（订阅、推荐、反馈闭环）
- 埋点与度量（检索成功率、节省时间、复用率、采纳率）
- 验收口径（MVP验收清单 + 边界条件）
